## Context

目前預覽流程由 Rust 端用 FFmpeg 將 RTSP 轉為 MJPEG，前端再用 img 元素讀取本機 /stream 端點。這個架構只適合顯示影像，無法承載音訊，也無法在切換來源時精確控制「舊畫面何時移除、新畫面何時顯示」，因此自動預覽切換攝影機時會出現連線中與舊畫面交錯的現象。

這次變更同時碰到前端狀態管理、預覽輸出格式與 FFmpeg 啟動方式，屬於跨前後端的預覽管線調整。目標不是單純補一個音量按鈕，而是把預覽輸出改成瀏覽器可直接播放影音、且能以明確就緒時機切換畫面的流程。

## Goals / Non-Goals

**Goals:**

- 讓預覽在來源串流包含音訊時可以播放聲音。
- 讓使用者可以在預覽畫面切換是否播放聲音，且切換時不需要重新建立來源連線。
- 切換攝影機或自動預覽切換來源時，先清除舊預覽，等新來源可播放後再顯示新畫面。
- 避免舊來源的非同步結果覆蓋新來源的預覽狀態或 URI。

**Non-Goals:**

- 不處理音量滑桿、錄音、雙向對講或音訊裝置選擇。
- 不新增多路同時預覽。
- 不變更攝影機搜尋、儲存或 ONVIF 認證流程。

## Decisions

### 1. 以可攜帶音訊的瀏覽器相容預覽格式取代純 MJPEG 輸出

現有 MJPEG 管線無法傳遞音訊，因此預覽輸出需要改成瀏覽器可直接播放影音的格式。採用本機 HTTP 提供的 HLS 預覽會比自製 MSE 管線或 WebRTC 更務實，因為 FFmpeg 可直接產生 playlist 與 segment，前端也能透過 video 元素與 HLS 播放器控制 ready 狀態與 muted 屬性。

替代方案：

- 維持 MJPEG 並額外開一條音訊串流：前端需要同步兩條獨立媒體，切換與對時更複雜。
- 自建 fragmented MP4 + MSE：延遲可較低，但實作與除錯成本高，超出本次變更必要範圍。

### 2. 在 StreamManager 中引入 preview session，避免舊來源結果殘留

目前 start 會先 stop 舊預覽，但前端仍保留舊 previewUrl，直到新 URL 回來前 img 還在畫面上。新設計在每次啟動預覽時建立新的 session 識別，Rust 端只暴露目前有效 session 的 manifest/segment；前端在切換來源時立即清空舊媒體元素，並只接受最新 session 的回應。

替代方案：

- 只在前端切換前先把 previewUrl 設為空：可以遮掉舊畫面，但無法防止舊請求晚回來後覆寫新狀態。
- 完全依賴 stop 完成再 start：仍無法解決非同步競態與就緒事件晚到的問題。

### 3. 用播放器就緒事件決定何時顯示新預覽

需求核心是切換來源後不要先閃回上一台畫面，因此畫面顯示時機應由新來源的可播放事件控制，而不是由 API 呼叫回傳時間控制。前端會在切換開始時進入「連接訊號中」狀態、隱藏舊畫面，等 video/HLS manifest 與第一段資料已可播放時才切換到新畫面。

替代方案：

- 只要拿到新的 preview URL 就立即顯示：若播放元件尚未載入完成，仍可能出現空白、殘影或上一個來源的 DOM 尚未釋放。

### 4. 音訊播放開關只控制前端播放狀態，不重新協商串流

是否播放聲音屬於本機播放行為，應由前端的 muted 屬性控制，而不是每次切換音訊開關都重啟 FFmpeg。若來源偵測不到音訊軌，前端維持靜音並禁用或隱藏音訊開關。

替代方案：

- 重新啟動帶音訊或無音訊的不同 FFmpeg 命令：切換成本高，也會再次造成預覽重連。

### 5. 在預覽建立期間偵測來源是否有音訊軌

為了讓 UI 只在有音訊的來源提供控制開關，預覽啟動結果需要帶回 hasAudio 類型的中繼資料。這個資訊可由 FFmpeg/ffprobe 在啟動時探測輸入流後提供給前端，一併作為 preview session 狀態的一部分。

替代方案：

- 永遠顯示音訊開關：行為簡單，但對沒有音訊的攝影機會造成誤導。

## Risks / Trade-offs

- [HLS 會增加一些播放延遲] → 以較短的 segment 與低延遲 FFmpeg 參數控制延遲，優先解決正確切換與音訊支援。
- [不同平台 WebView 對 HLS 支援程度不同] → 前端採播放器封裝並在必要時加入 hls.js，避免只依賴原生支援。
- [FFmpeg/ffprobe 啟動與探測成本增加] → 將探測結果綁定在 preview session 建立流程，避免重複探測。
- [切換過程中的舊請求可能晚回來] → 以 session ID 或 request token 比對，只允許最新請求更新 store 狀態。
